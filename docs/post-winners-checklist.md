# HireProof Post-Winners Checklist

Use this after hackathon results are announced. Keep this local until public updates are safe.

## First Hour

- Save the final winner/finalist/not-selected result screenshot.
- Archive the submission page, release page, live demo, package page, and GitHub repo state.
- Record the exact result wording and announcement URL.
- Do not rewrite the repo story immediately; update the status once and keep the proof stable.

## If HireProof Wins Or Places

- Update README hero copy with the result.
- Add a short result note to the v1.0 release description.
- Post the LinkedIn launch draft with the result in the first paragraph.
- Add the result to portfolio case studies.
- Create one visual post using `public/social/github-social-preview-1280x640.png`.
- Add a proof screenshot to `docs/final-release-proof.md` or a new public proof note.
- Consider a small v1.0.1 release only for documentation/status updates, not rushed product changes.

Suggested line:

> HireProof placed in the Cursor Hackathon: a solo one-week build that turned suspicious job-post verification into a proof-backed AI agent workflow.

## If HireProof Is Not Selected

- Keep the portfolio story positive and outcome-independent.
- Do not apologize for the result.
- Lead with what shipped: production app, proof-backed audit flow, release assets, packages, docs, and multi-surface agent coverage.
- Post the portfolio-focused LinkedIn version instead of winner wording.
- Add a short "Built and shipped for Cursor Hackathon" note, not a negative result note.

Suggested line:

> HireProof shipped as a production-facing Cursor Hackathon build. The community-vote format meant placement was not fully within the product team's control, but the controllable work is clear: app, API, proof pages, docs, and a pilot path.

## Product Follow-Up

- Keep public live usage cost-capped with the after-hackathon guardrails.
- Keep BYOK as the serious-usage path for live model/search spend.
- Recheck `/api/health`, `/audit`, `/docs`, and the v1.0 release page before sharing widely.
- Recapture one fresh product screenshot after any public status update.
- Avoid changing the core positioning away from job scams unless there is a deliberate post-hackathon product decision.

## Product Design Follow-Up

- Turn `docs/portfolio-product-design-case-study.md` into a website case study.
- Add a product decision timeline: problem, wedge, evidence model, trust UX, distribution surfaces, guardrails.
- Include "what I cut" to show judgment: no generic fraud platform pivot, no hidden ML claims, no unsupported deepfake claim.
- Add one metric or proof point when real usage exists: audit count, package downloads, stars, demo visits, or recruiter/client interest.

## UI/UX Follow-Up

- Build a visual case-study page using the live product hero and result screen.
- Show three screens only: input, investigation/report, proof/export.
- Add responsive proof screenshots once they are captured after the competition.
- Keep the design story focused on trust, readability, and fast decision-making.

## Developer Portfolio Follow-Up

- Link to packages and release assets.
- Add a concise architecture diagram section using `public/technical_architecture_fraud_v2.png`.
- Mention verified boundaries: demo mode is seeded, live mode depends on credentials, WDK proof is accepted-run unless completed-result proof is captured.
- Add a "how I would productionize next" section: account auth, billing/BYOK, abuse protection, provider budgets, durable workflow timeline, and marketplace review.

## Public Update Order

1. Confirm live app and release links still work.
2. Update portfolio case study locally.
3. Update README/result copy if the result is public and useful.
4. Post LinkedIn.
5. Post X/Twitter.
6. Share in relevant builder or career communities.
7. Only after the above, decide whether to tag a docs-only release.

## Do Not Do Immediately

- Do not push rushed feature changes just to react to results.
- Do not remove cost guardrails for public live usage.
- Do not claim broad fraud coverage.
- Do not claim live Discord proof or completed WDK workflow proof unless captured.
- Do not publish private logs, raw secrets, or screenshots with sensitive account data.
