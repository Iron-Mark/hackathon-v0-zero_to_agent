# HireProof Post-Hackathon Startup Plan

Last updated: 2026-05-10, Asia/Manila

HireProof shipped as a complete, production-facing Cursor Hackathon build. The final ranking was shaped by factors outside the product itself, including community-vote reach and distribution timing, so the right post-hackathon read is "validated a serious product direction and moved into pilot mode."

The next phase should keep the proof stable, protect provider spend, and validate whether real users want this as a community safety tool, developer API, or hiring-page trust signal.

## Operating Position

- Keep the product centered on suspicious job posts, recruiter messages, screenshots, and apply links.
- Do not pivot into a generic fraud or security platform without pilot evidence.
- Keep public demo paths live for evaluation.
- Move serious live provider usage to BYOK-first controls.
- Treat external marketplace approvals, Chrome Web Store publication, Discord proof, and completed WDK transcripts as proof-gated milestones.
- Frame the hackathon result as outcome-independent: the controllable work was shipped, verified, deployed, and packaged; community-vote placement was not a pure product-quality measure.

## 90-Day Plan

1. Stabilize production and cost posture.
   - Verify `https://hireproof.tech`, `/api/health`, `/audit`, `/developer`, and `/docs/pilot`.
   - Set post-hackathon provider flags so public demo stays available without open-ended live provider spend.
   - Keep the Developer Portal as the entry point for API keys, usage, hosted BYOK, webhooks, and verified domains.

2. Package a pilot offer.
   - Target career communities, schools, bootcamps, recruiters, and job boards.
   - Offer three paths: free demo checks, BYOK live checks, and organization/API pilot access.
   - Capture real workflows: suspicious post review, community chat screening, verified hiring page, and exportable report handoff.

3. Validate demand before adding billing.
   - Look for repeat use, trusted referrals, false-positive feedback, API/webhook demand, and willingness to bring provider keys.
   - Avoid building payments until at least one pilot group shows repeated usage.
   - Keep pricing copy as suggested packaging, not a live paid checkout claim.

4. Decide the startup direction.
   - Community SaaS if job-seeker groups use it repeatedly.
   - B2B trust layer if employers or job boards care about verified hiring pages and badge proof.
   - Developer/API product if integrations and agent workflows become the strongest signal.

## Immediate Acceptance Checks

- Public demo still works.
- `/api/health` reports the intended cost posture.
- BYOK and API-key docs match the actual Developer Portal behavior.
- Public pages do not claim a hackathon win, marketplace approval, or unproven ML capabilities.
